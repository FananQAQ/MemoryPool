## 思考题参考答案

### 1. 为什么 `block_size_` 不能太小？

因为空闲 block 在内存池内部要被临时当成 `FreeNode` 使用，而 `FreeNode` 至少需要存下一个指针 `next`。

这意味着每个 block 至少要有足够空间容纳：

- 一个 `FreeNode*`

如果 `block_size_` 小于一个指针大小，那么空闲块连最基本的链表管理信息都放不下，free list 就无法成立。

所以这个 lab 里会对 `block_size_` 做一个最小值保护，本质上是在保证：

- 每个空闲块都能被安全地解释为一个 `FreeNode`

### 2. 为什么 block 起始地址可以直接复用为 `FreeNode`？

因为 free list 的节点不是额外单独分配出来的，而是直接借用“空闲 block 自己的起始内存”。

当一个 block 处于空闲状态时，它的前几个字节可以这样使用：

- 用来存 `FreeNode::next`

也就是说，这块内存虽然本质上还是一个 block，但在“空闲状态”下，可以把它的起始地址解释成 `FreeNode*`。

这样做的好处是：

- 不需要额外为链表节点申请内存
- 减少管理开销
- 分配和释放都可以保持 O(1)

### 3. 如果用户传入了不是块起点的指针，为什么要拒绝？

因为内存池是按“整个 block”为单位管理的，不是按 block 内部任意地址管理的。

如果传进来的指针只是某个 block 中间的位置，而你仍然把它当成一个空闲块挂回 free list，就会导致：

- free list 节点地址错误
- 下次分配得到的地址不是合法 block 起点
- 内存池内部结构被破坏
- 进一步引发内存覆盖、非法访问甚至崩溃

所以释放时必须保证两件事：

- 这个指针确实属于当前内存池
- 这个指针正好落在某个 block 的起始地址上

### 4. 如果同一个指针被重复释放，会发生什么？

这就是典型的 double free 问题。

如果不做检测，重复释放通常会让同一块内存被多次插入 free list，后果可能包括：

- 同一地址在 free list 中出现多次
- 多次 `allocate()` 拿到同一块内存
- 两个对象实际上共享同一地址
- 写入一个对象时覆盖另一个对象的数据
- 严重时导致链表损坏、程序行为异常甚至崩溃

教学版 lab 可以先不处理这个问题，但在更正式的实现中，通常会考虑加入 double free 检测。

### 5. 如果想支持不同大小的对象池，应该怎么设计？

一个常见做法是准备多个不同 block size 的池，也就是所谓的 size class。

例如：

- 16 字节池
- 32 字节池
- 64 字节池
- 128 字节池

分配时根据请求大小选择最合适的池：

- 请求 12 字节，走 16 字节池
- 请求 20 字节，走 32 字节池
- 请求 70 字节，走 128 字节池

这样做的优点是：

- 仍然可以保留固定块池分配快的特点
- 能支持多种大小的对象

这个设计真正的难点在于释放时如何判断一个指针属于哪个池。常见做法包括：

- 遍历各个池并调用 `owns(ptr)`
- 在分配时记录额外元数据
- 通过 chunk 信息反查所属池

这也是从“单一固定块池”继续进阶到“更真实的内存分配器”的关键一步。
